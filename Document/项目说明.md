# Landsong 项目说明

## 项目定位

Landsong 是一个网格放置建造游戏。玩家在地图网格上规划建筑、道路与功能区，通过采集、生产、消耗和扩建形成持续发展的基地或聚落。

当前项目已完成基础骨架与运行入口，优先目标是先完成一个可玩的垂直切片：玩家可以查看网格、选择建筑、预览占位、合法放置、消耗资源、保存并重新加载已有建筑。

建筑系统已在 2026-07-13 切换到最终家族架构。建筑相关交接统一从 [Document/建筑系统/README.md](建筑系统/README.md) 进入；下列阶段描述不再允许按“每等级一个 Prefab”理解。

## 基础约束

1. 项目中的资源加载使用 Addressables。
2. 项目输入使用 Unity 新版 Input System，不使用旧版 `UnityEngine.Input`。
3. 项目将来需要同时支持 PC 与移动平台，因此输入逻辑要兼容鼠标、触摸与后续 UI 操作。
4. 核心玩法数据优先使用 ScriptableObject 配置，运行时只保存实例状态。
5. 网格使用 1:0.5 的等距菱形网格；网格坐标、占用、放置规则属于玩法层；高亮、预览、特效属于表现层。
6. 先做单机本地存档，不提前设计联机或多人同步。
7. 优先做可验证的小闭环，每个阶段都要能在 Unity 编辑器中手动运行检查。

## 推荐目录规划

```text
Assets/Landsong/
  Art/                 美术资源
  Audio/               音频资源
  Data/                ScriptableObject 配置
  Prefabs/             可 Addressables 加载的预制体
  Scenes/              游戏场景
  Scripts/
    Core/              启动、服务注册、通用基础设施
    Grid/              网格坐标、格子数据、占用查询
    Placement/         放置输入、预览、合法性检测
    Building/          建筑定义、建筑实例、生产逻辑
    Economy/           资源库存、消耗、产出
    Save/              存档数据、序列化、加载恢复
    UI/                HUD、建造菜单、提示面板
  Settings/            项目内配置资源
```

## 开发步骤

### 阶段 1：项目骨架与运行入口

目标：建立最小可运行场景和代码结构。

状态：已完成。

1. 创建主场景 `Game`，放置主相机、灯光、事件系统和游戏入口对象。
2. 建立 `Assets/Landsong` 下的目录结构。
3. 创建 `GameBootstrap`，负责初始化网格、资源库存、建造系统和 UI。
4. 配置 Addressables 分组，先放入少量测试建筑预制体。
5. 确定输入方案：鼠标选择格子、左键确认、右键取消、滚轮缩放或相机移动。

验收标准：

- 进入 Play Mode 后可以看到一个空场景。
- 控制台没有启动错误。
- Addressables 可以加载一个测试预制体。

### 阶段 2：网格系统

目标：让游戏拥有稳定的 1:0.5 等距菱形格子坐标、世界坐标转换和占用记录。

状态：代码已实现，等待场景挂载与 Unity Play Mode 验证。

1. 实现 `GridPosition`，表示整数网格坐标。
2. 实现 `GridLayoutService`，提供 1:0.5 等距菱形网格坐标与世界坐标互转。
3. 实现 `GridMap`，记录格子是否可用、是否被占用、占用者是谁。
4. 做一个 `GridDebugRenderer` 或调试 Gizmos，能在场景中看到网格线。
5. 支持建筑占用多格，例如 `1x1`、`2x2`、`3x2`。

验收标准：

- 鼠标或触摸移动时可以稳定命中对应菱形格子。
- 多格范围计算正确。
- 被占用格子不会被重复占用。

### 阶段 3：建筑定义与 Addressables 加载

目标：用数据驱动建筑，而不是在代码里写死建筑类型。

1. 每个家族创建一个 `BuildingFamilyDefinition`，其中的 `BuildingDefinition` 保存 FamilyId、固定占地和放置规则。
2. 施工回合、连续运营等级、目标级科技条件/费用和等级数值保存在 FamilyDefinition。
3. 可复用能力保存在独立 `BuildingModuleSetDefinition`，表现映射保存在 `BuildingPresentationDefinition`。
4. `BuildingCatalog` 只登记家族；每个家族只引用一个轻量 Runtime Prefab。
5. 施工、等级和样式 View 独立于 Runtime Prefab，可按需改为 Addressable 加载。
   
验收标准：

- UI 或测试快捷键可以选择不同建筑定义。
- 每个 FamilyId 只对应一个 Runtime Prefab，等级不创建新 Prefab/脚本。
- 新增通用数据型建筑时不需要修改放置代码；家族独有玩法只写在一个不带等级后缀的脚本中。
- 缺少等级美术时使用同样式低等级或统一占位，不阻塞玩法数据。

### 阶段 4：放置预览与合法性检测

目标：玩家能看到建筑将要放在哪里，以及为什么能放或不能放。

1. 实现 `PlacementController`，处理当前选中的建筑和玩家输入。
2. 实现半透明预览对象，跟随鼠标所在格子移动。
3. 根据占用、地图边界、建筑尺寸、资源是否足够判断能否放置。
4. 用颜色区分合法与非法预览。
5. 支持取消放置和连续放置。

验收标准：

- 合法位置显示可放置状态。
- 已占用位置、越界位置和资源不足时不能确认。
- 成功放置后网格占用立即更新。

### 阶段 5：资源与建造消耗

目标：建立最小经济闭环，让建造行为有成本。

1. 库存统一使用 `InventoryService` / `ItemDefinition`。
2. `BuildingDefinition.PlacementCosts` 在确认放置时一次性扣除。
3. `BuildingFamilyDefinition.Construction` 的第 N 项对应第 N 个施工回合；有费用时必须连接 Resource 提供点并原子扣除，成功后同一实例推进进度。
4. 施工完成后同一实例切换为 `Operational/LV1`，不得替换 Prefab。
5. UI 分别显示放置费用、施工总费用和当前施工回合需求。

验收标准：

- 资源不足时无法建造。
- 建造过程中每个施工回合会扣除对应资源。
- UI 资源显示与实际库存一致。

### 阶段 6：建筑运行逻辑

目标：让建筑不只是静态模型，而能参与生产或功能系统。

1. `BuildingBase` 记录稳定实例身份、Stage、Level、Style 和施工进度，静态数据来自 FamilyDefinition。
2. 岗位、生产、加工、库存、作物等复用能力由带稳定 ModuleId 的模块唯一持有状态。
3. 家族脚本只实现居民房、市场等无法泛化的领域流程，同一脚本覆盖全部等级。
4. 原地升级统一走 `BuildingUpgradeService`，保留模块和家族状态。
5. 道路、资源连接和空间效果继续使用通用查询/模块接口。
6. 建筑详情 UI 从家族、等级配置、模块和家族状态读取信息。

验收标准：

- 至少一个建筑可以周期产出资源。
- 建筑状态能在 UI 中查看。
- 删除或拆除建筑后占用格子会释放。

### 阶段 7：存档与读取

目标：可以保存玩家已建造的世界并恢复。

1. 设计 `SaveGameData`，保存资源库存、建筑实例列表和必要的全局状态。
2. 建筑实例保存 `FamilyId + InstanceId + Stage + Level + StyleId + ConstructionProgress`、网格坐标、家族状态和按 ModuleId 保存的模块状态，不保存 Prefab/View 引用。
3. 加载时通过 `BuildingCatalog` 找到家族唯一 Runtime Prefab，再恢复身份、配置和表现。
4. 支持手动保存、手动读取和退出前保存。
5. 可保留存档版本号，但本轮架构明确不兼容旧等级 Prefab 存档，不增加旧 API/数据迁移分支。

验收标准：

- 放置多个建筑后保存，重进 Play Mode 可以恢复。
- 存档中没有 Unity 对象引用。
- 删除建筑、资源变化等状态能正确恢复。

### 阶段 8：建造 UI 与操作体验

目标：把测试快捷键替换成正式可用的建造界面。

1. 实现建造菜单，按分类展示建筑。
2. 每个建筑按钮显示名称、图标和成本。
3. 资源不足的建筑显示不可用状态。
4. 添加当前选中建筑、取消放置、拆除模式等操作。
5. 补充鼠标悬停提示和错误提示，例如“空间不足”“资源不足”。

验收标准：

- 玩家不依赖调试快捷键也能完成建造流程。
- UI 状态与放置系统状态同步。
- 错误提示清晰且不会刷屏。

### 阶段 9：地图与关卡扩展

目标：从单一空地扩展到有地形约束和可规划空间的地图。

1. 定义地图尺寸、可建造区域和不可建造区域。
2. 添加地形类型，例如草地、水面、岩石、道路。
3. 放置规则支持指定建筑只能放在某些地形上。
4. 支持地图初始化配置，后续可扩展为随机地图或关卡配置。
5. 增加相机边界，避免玩家移出地图。

验收标准：

- 建筑会根据地形规则判断是否合法。
- 地图边界和不可建造区域生效。
- 地图配置可以被替换或扩展。

### 阶段 10：垂直切片打磨

目标：整理成一个完整、稳定、可演示的小版本。

1. 完成一组核心建筑：资源采集、资源生产、仓储、道路或装饰。
2. 补充基础音效、放置反馈、选择反馈和错误反馈。
3. 整理 Addressables 分组和资源命名。
4. 增加基本测试：网格坐标转换、占用检测、资源扣除、存档数据转换。
5. 做一次从新游戏到保存读取的完整手动验收。

验收标准：

- 玩家可以连续游玩 10 到 15 分钟。
- 核心建造、资源、存档流程没有明显阻断问题。
- 后续新增建筑和地图规则有清晰扩展点。

## 优先级建议

1. 先做网格与放置，不先做完整资源链。
2. 先做数据结构和 Addressables 加载方式，不先堆具体建筑数量。
3. 先做本地单机闭环，不提前引入联机、复杂 AI 或大规模自动化。
4. 先把调试工具做好，例如网格显示、占用显示、当前鼠标格子显示。
5. 每完成一个阶段，都保留一个可以运行的稳定版本。

## 第一轮最小可玩目标

第一轮开发建议只包含以下内容：

1. 一个固定尺寸地图。
2. 三种建筑：仓库、资源生产建筑、道路。
3. 一套资源：木材。
4. 网格放置、占用检测、资源扣除。
5. 保存和读取建筑列表。

完成这一轮后，再继续扩展更多资源、生产链、建筑功能和地图规则。

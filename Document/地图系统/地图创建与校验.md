# 开发：创建地图

> 本文是策划/编辑器操作指南。需要修改地图代码、格子规则、加载流程、Overlay 或建筑连接时，先阅读 [地图系统](README.md)。

## 1. 地图资产的最终结构

Landsong 中的一张地图不再是一个完整地图 Prefab，而是下列资产的组合：

- 一个 **Content Scene**：保存该地图的 Grid、静态 Tilemap、地形层和初始建筑可视化模板。
- 一个 **MapDefinition**：保存稳定地图 ID、显示名称、图标、描述和 Content Scene 引用。
- 一个全项目共享的 **MapCatalog**：登记所有 MapDefinition。
- Game Scene 中的公共 **MapRuntimeHost**：加载 Content Scene 后注入格子规则、占用数据、Overlay 和运行时建筑。

> 创建新地图时，不要复制 Game Scene、相机、UI 或 GameSystem。每张地图只增加 Content Scene 和 MapDefinition。

## 2. 一次性项目前置检查

这部分全项目只需配置一次。如果已经完成，直接进入第 3 节。

### 2.1 Game Scene

Game Scene 中必须有且只有一个 `MapRuntimeHost`，并绑定。当前 `Game.unity` 已完成以下公共配置，创建普通新地图时不要重复创建或在 Content Scene 中复制：

- `Grid Map`：Game Scene 中的 `GridMapBehaviour`。
- `Rule Set`：全项目唯一的 `GridRuleSet`。
- `Overlay Service`：Game Scene 中的 `GridOverlayService`。
- `Runtime Buildings Root`：可选。不绑定时，运行时会自动创建 `Runtime Buildings`。

Game Scene 中的 `GridMapBehaviour` 只负责运行时占用数据；新地图的 Base/Terrain Tilemap 由 Content Scene 加载后注入。

### 2.2 全局规则与地图目录

- 项目中应只有一个正式 `GridRuleSet`。
- `GridRuleSet` 当前控制默认地形、默认是否可建造，以及普通格/道路/高级道路的行动力消耗。
- `Assets/Landsong/Objects/SO/MapCatalog.asset` 必须保持 Addressable，Address 为 `MapCatalog`。
- `AppManager` 的地图目录 Addressables Key 应为 `MapCatalog`。

### 2.3 Grid 对齐

Game Scene 中的运行时 Grid 与每张 Content Scene 的 Grid 必须保持一致：

- Cell Size
- Cell Gap
- Cell Layout
- Cell Swizzle
- Transform 原点与旋转

否则运行时占用层、建筑与静态地图可能发生偏移。项目不支持建筑旋转。

## 3. 创建 Content Scene

### 3.1 新建 Scene

1. 在 Unity 中新建一个空 Scene。
2. 建议保存到统一地图目录，例如：

   `Assets/Landsong/Scenes/MapScenes/Map_RiverValley.unity`

3. 不要在 Content Scene 中放置下列运行时对象：

   - Camera
   - UI / EventSystem
   - GameSystem
   - `MapRuntimeHost`
   - `GridMapBehaviour`
   - Occupancy Tilemap
   - Overlay Tilemap

推荐的 Scene 层级：

```text
MapContentRoot                         [MapContentAuthoring]
├─ Grid                              [UnityEngine.Grid]
│  ├─ Base Tilemap                 [Tilemap + TilemapRenderer]
│  ├─ Water Tilemap                [Tilemap + TilemapRenderer]
│  ├─ Obstacle Tilemap             [Tilemap + TilemapRenderer]
│  ├─ Road Tilemap                 [可选，登记到 Terrain Layers]
│  └─ Decoration Tilemap           [可选，不登记，纯视觉]
└─ Initial Buildings
   ├─ PlayerHome                      [建筑 Prefab Instance]
   └─ ...                             [建筑 Prefab Instance]
```

`Initial Buildings` 只是用于整理层级的空父物体，不需要挂载任何初始建筑组件。它和所有初始建筑必须位于 `MapContentAuthoring` 层级下。

### 3.2 绘制 Base Tilemap

选中 `MapContentAuthoring` 后，可以点击“一键生成基础 Tilemap”。该按钮会：

- 复用当前 Scene 中唯一的 Grid；如果没有 Grid，则按项目的等距网格参数创建一个。
- 创建或复用 `Base Tilemap`，并自动绑定。Base 同时承担地表视觉和逻辑边界，不再创建独立的陆地 Tilemap。
- 创建或复用 `水域 Tilemap` 和 `障碍 Tilemap`。
- 自动将水域与障碍以 `Replace Default Terrain Key = true` 登记到 Terrain Layers，无需再手动添加。

`Terrain Layers` 是水域、障碍和道路等逻辑地形的唯一数据源。`MapContentAuthoring` 不再另外序列化一份水域/障碍 Tilemap 引用；一键生成按钮会直接查找和更新数组中对应 Key 的项。

如果 Scene 中存在多个 Grid，按钮不会自行猜测；需要先手动绑定 `MapContentAuthoring.Unity Grid`。

`Base Tilemap` 不仅是背景，它定义地图的逻辑边界：

- 有 Base Tile 的格子才属于地图。
- 没有 Base Tile 的格子被视为越界。
- 空的 Base Tilemap 会使地图校验和加载失败。

先完整画出所有可玩格子，再绘制水域、道路等地形层。

> 当前的“默认可建造”是全局规则，还没有独立的逐格禁止建造层。需要完全排除出地图的格子时，不要在 Base Tilemap 中绘制它。

### 3.3 绘制地形 Tilemap

未被主地形层替换的 Base 格子，使用 `GridRuleSet.DefaultTerrainKey`，当前通常为 `陆地`。

`MapContentAuthoring.Terrain Layers` 中每一项包含：

- `Key`：逻辑地形名称。已有 Key 为 `陆地`、`水域`、`道路`、`高级道路`。
- `Tilemap`：对应的地形 Tilemap。
- `Replace Default Terrain Key`：该层是替换主地形，还是叠加一个额外标签。

建议配置：

| 地形 | Replace Default Terrain Key | 含义 |
| --- | --- | --- |
| 水域 | `true` | 该格从陆地替换为水域，普通陆地建筑不能放置 |
| 障碍 | `true` | 该格仍保留在 Base 地图内，但主地形替换为障碍，陆地建筑不能放置 |
| 道路 | `false` | 该格仍是陆地，同时拥有道路标签 |
| 高级道路 | `false` | 该格仍是陆地，同时拥有高级道路标签 |

注意：

- 主地形层的先后顺序会影响重叠格子，后面的“替换默认地形”层会成为最终主地形。
- 应尽量避免多个主地形层在同一格重叠。
- 纯装饰 Tilemap 无需登记到 `Terrain Layers`；未登记的 Tilemap 不参与放置或寻路规则。
- 建筑允许的地形由建筑 `BuildingDefinition.Required Terrain Keys` 决定。一个多格建筑的所有占地格都必须满足要求。

障碍层的推荐做法：

1. Base Tilemap 继续在障碍下方保留 Tile，因此这些格仍属于地图，不会变成越界空洞。
2. 在 `障碍 Tilemap` 上用不同瓦片表达山体、岩石、遗迹或其他不可建造物。
3. 一键生成按钮会自动登记：`Key = 障碍`、`Replace Default Terrain Key = true`。
4. 普通建筑默认要求 `陆地`，障碍格的最终主地形是 `障碍`，因此放置检查会返回 `TerrainMismatch`。

> 当前“障碍”配置只保证不能放置建筑。地形寻路目前只支持道路等低成本标签，还不支持“完全不可通行”的静态地形规则。如果障碍还应阻断资源连接路径或其他移动，需要再为 `GridRuleSet` 增加逐地形 `Traversable` 规则。

换句话说，当前山体画在障碍层后：普通陆地建筑不能建在山体格上，但 `GridMapBehaviour.CanTraverse(...)` 仍只检查 Base 边界和运行时建筑占用，资源连接路径可能穿过山体。这是已知能力边界，不要把“不能建造”误写成“不可通行”。

### 3.4 绑定 MapContentAuthoring

在 `MapContentRoot` 上添加 `MapContentAuthoring`，并绑定：

1. `Unity Grid`：Scene 中的 Grid。
2. `Base Tilemap`：已经绘制且非空的 Base Tilemap。
3. `Terrain Layers`：所有需要参与逻辑规则的地形 Tilemap。

每张 Content Scene 必须有且只有一个 `MapContentAuthoring`。

### 3.5 MapContentAuthoring 与 GridMapBehaviour 的 Tilemap 关系

两者不是两套独立地图：

| 所属 | Tilemap | 职责 |
| --- | --- | --- |
| Content Scene / `MapContentAuthoring` | Base Tilemap | 地图逻辑边界；没有 Base Tile 就是越界 |
| Content Scene / `MapContentAuthoring` | Terrain Layers | 水域、道路等静态地形标签与行动力规则 |
| Content Scene | 未登记的其他 Tilemap | 纯视觉，不参与放置和寻路 |
| Game Scene / `GridMapBehaviour` | Occupancy Tilemap | 可选的运行时占用可视化；真正占用数据保存在 `GridMapBehaviour` 字典中 |

地图加载后，`MapRuntimeHost` 会把 `MapContentAuthoring` 的 Grid、Base Tilemap 和 Terrain Layers 注入 `GridMapBehaviour`。因此 `GridMapBehaviour` Inspector 不再单独绑定 Base/Terrain/RuleSet；它只保留 Occupancy Tilemap 配置。

Overlay Tilemap 也不属于 Content Scene。`GridOverlayService` 会在运行时按已使用的 `GridOverlayChannelDefinition` 动态创建并挂到 Content Grid；卸载地图时统一清理。

所谓“不匹配”分为两种：

- **Content 逻辑层绑错**：例如视觉上画了水，但 Water Tilemap 没有登记到 Terrain Layers。游戏会把该格当作默认陆地，出现“看起来是水、实际能放陆地建筑”的错误。
- **Game Scene Grid/Occupancy 与 Content Grid 几何参数不匹配**：如果继续运行，Occupancy Tilemap 会偏移、缩放或形状错位。当前实现会在 `MapRuntimeHost.TryBind` 阶段直接拒绝加载并返回明确错误，不允许地图带着这种错位进入游戏。

所以必须确保 Game Scene Grid 与 Content Grid 的 Cell Size、Cell Gap、Cell Layout、Cell Swizzle 和 Transform 一致，同时保证所有会影响逻辑的 Tilemap 都正确登记在 `MapContentAuthoring`。

## 4. 配置初始建筑

初始建筑由 `MapContentAuthoring` 直接统一管理：

1. 在 `MapContentAuthoring` 层级下创建 `Initial Buildings` 空 GameObject。
2. 将需要的建筑 Prefab 直接拖入其子级，在 Scene 中拖动到期望位置。
3. 选中 `MapContentAuthoring`，点击“吸附初始建筑到格子”。
4. 通过 Scene Gizmo 检查每个建筑的完整占地。

Gizmo 颜色：

- **青色半透明**：建筑视觉中心已与实际占地对齐。
- **橙色**：建筑预览与实际占地不对齐；需要执行吸附，否则地图校验不会通过。

重要规则：

- 不需要为初始建筑或 `Initial Buildings` 父物体添加任何辅助组件；`MapContentAuthoring` 会自动扫描它层级下的全部 `BuildingBase`。
- 子级 Prefab Instance 只是策划预览与生成模板。进入运行时后会自动隐藏，不会直接注册成真实建筑。
- 真正初始建筑只在“新游戏首次创建地图”时由 `BuildingService` 生成；读档时只使用存档快照。
- 建筑根 Transform 不允许旋转；系统中没有建筑旋转概念。
- 占地不能越出 Base Tilemap、不能互相重叠，并且必须满足建筑地形要求。
- 所有模板会先统一预检。任何一个失败都会取消整张地图的初始建筑创建，不会留下半完成状态。

地图需要默认家园或资源提供点时，应将已配置好提供能力的 PlayerHome Prefab 直接放到 `MapContentAuthoring` 层级下。

## 5. 创建 MapDefinition

1. 在 Project 窗口中右键：

   `Create > Landsong > Map > Map Definition`

2. 建议保存到：

   `Assets/Landsong/Objects/SO/MapDef/`

3. 填写字段：

| 字段 | 要求 |
| --- | --- |
| Stable Map ID | 必填且全项目唯一，建议使用 `map_river_valley_01` 这类稳定英文 ID |
| Display Name | 给玩家看的名称，可以修改，不应用来当存档标识 |
| Icon | 地图选择界面图标 |
| Description | 地图介绍 |
| Addressable Content Scene | 第 3 节创建的 Content Scene |

`MapId` 登记后不要因为显示名称变化而修改，也不允许两张地图使用相同 ID。

## 6. 将 Content Scene 设为 Addressable

1. 保存 Content Scene。
2. 在 Addressables Groups 窗口中将该 Scene 加入一个地图资产 Group；可以新建 `Maps` Group 统一管理。
3. 确认 Scene 已是 Addressable Asset。Address 建议使用稳定名称，例如 `Map/map_river_valley_01`。
4. 回到 MapDefinition，将该 Scene 绑定到 `Addressable Content Scene`。

Content Scene 由地图加载器以 Additive 模式加载，不要让其取代 Game Scene，也不要在普通流程中直接跳转到该 Scene。

## 7. 登记到 MapCatalog

推荐使用项目内的地图工具：

1. 打开 `Landsong > 地图 > 地图目录与校验`。
2. `Map Catalog` 绑定 `Assets/Landsong/Objects/SO/MapCatalog.asset`。
3. `Map Definition` 绑定刚创建的 MapDefinition。
4. 点击“登记到 Catalog”。
5. 选中 MapCatalog，确认地图已出现在 `Maps` 列表中。

如果所有 MapDefinition 都保存在 Catalog 配置的 `MapDefinition 目录`，也可在 MapCatalog Inspector 中使用“从目录登记 MapDefinition”批量刷新。

## 8. 校验与测试

### 8.1 编辑器校验

在 `Landsong > 地图 > 地图目录与校验` 窗口中：

1. 绑定 MapDefinition。
2. 点击“校验当前地图”。
3. Console 应输出 `地图校验通过：<mapId>`。

当前编辑器校验会检查：

- MapDefinition 的 MapId 和 Content Scene 是否有效。
- Scene 中是否恰好有一个 `MapContentAuthoring`。
- Grid/Base Tilemap 是否绑定，Base Tilemap 是否非空。
- MapContentAuthoring 层级下的初始建筑是否都有有效的 Definition，以及预览是否已吸附。
- 初始建筑是否越出 Base Tilemap或相互重叠。

### 8.2 从当前地图开始 Play

1. 先保存 Scene 和所有资产。
2. 确保 MapDefinition 已登记到 MapCatalog。
3. 在地图工具中点击“从当前地图开始 Play”。

该按钮会从 `Start.unity` 进入正常启动流程，然后使用当前 MapId 创建新游戏。不要通过直接在 Content Scene 点击普通 Play 来替代这个流程。

### 8.3 运行时验收

至少检查以下内容：

- Content Scene 能够 Additive 加载，地图与运行时 Grid 没有偏移。
- 初始建筑数量、占地和地形正确。
- 新游戏生成后立即保存，退出再读档后不会重复生成初始建筑。
- 陆地/水域建筑的放置合法性正确。
- 道路与高级道路行动力消耗正确。
- 建筑红/绿占地、资源连接路径、Buff 范围等 Overlay 没有层级冲突。
- 地图能在主菜单地图选择中显示正确名称、图标和描述。

> 编辑器的静态校验不会覆盖所有运行时规则。例如初始建筑的完整地形要求会在新游戏创建时再检查，因此“从当前地图开始 Play”是必须步骤。

## 9. 常见问题

### 地图校验提示 Content Scene 无效

检查 Scene 是否已保存、是否设为 Addressable，以及 MapDefinition 的 `Addressable Content Scene` 是否绑定了该 Scene。

### 地图加载后为空或无法放置建筑

检查 `Base Tilemap` 是否实际画了 Tile，以及 `MapContentAuthoring.Base Tilemap` 是否绑到正确层。

### 建筑被误判为不满足地形要求

检查建筑的 `Required Terrain Keys`、Terrain Layer 的 `Key` 和 `Replace Default Terrain Key`。水域通常是替换主地形，道路通常是附加标签。

### 初始建筑没有出现

确认建筑 Prefab Instance 位于 `MapContentAuthoring` 层级下，已执行吸附，并且测试的是新游戏而不是已有存档。

### 障碍上不能建造，但资源路径仍然穿过

这是当前规则：障碍是替换主地形的“不可建造地形”，不是不可通行地形。资源寻路只把 Base 外和阻挡型运行时建筑视为不可通行。如果需要山体阻断寻路，应扩展共享 `GridRuleSet` 的逐地形可通行规则，并回归所有地图和资源连接预览。

### 放置普通建筑时没有资源或 Buff 范围

这通常不是地图配置问题。附加范围只在建筑声明相应能力时生成：Resource 消费者需要实现连接消费者接口或挂载消费者模块；Buff 建筑需要挂载 `BM_空间效果源` 并绑定有效的 `BuildingSpatialEffectDefinition`。详细规则见 [新增建筑与编辑器](../建筑系统/新增建筑与编辑器.md)。

### 点击“从当前地图开始 Play”后报告 MapId 不存在

先点击“登记到 Catalog”，并确认 MapCatalog 中没有重复 MapId。

### Addressables 在编辑器可用，但构建后加载失败

重新构建 Addressables Content，并检查 Content Scene 和 MapCatalog 是否都被包含在当前 Addressables Profile 中。

## 10. 新地图完成清单

- [ ] Content Scene 已保存。
- [ ] Scene 中恰好有一个 `MapContentAuthoring`。
- [ ] Grid 与 Game Scene 运行时 Grid 配置一致。
- [ ] Base Tilemap 已画满所有可玩格子。
- [ ] 水域/道路/高级道路等逻辑层已登记到 Terrain Layers。
- [ ] 纯视觉 Tilemap 没有误登记为地形层。
- [ ] 所有初始建筑 Prefab Instance 都位于 MapContentAuthoring 层级下。
- [ ] 已执行“吸附初始建筑到格子”，所有占地 Gizmo 都是青色。
- [ ] 初始建筑没有越界、重叠或地形错误。
- [ ] Content Scene 已设为 Addressable。
- [ ] MapDefinition 的 MapId 唯一，显示信息和 Scene 引用完整。
- [ ] MapDefinition 已登记到 MapCatalog。
- [ ] “校验当前地图”通过。
- [ ] “从当前地图开始 Play”验收通过。
- [ ] 构建游戏前已重新构建 Addressables Content。

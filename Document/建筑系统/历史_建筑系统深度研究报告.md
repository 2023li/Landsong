## 历史发现与当前处置

| 历史发现 | 处置结果 |
| --- | --- |
| 等级用不同 Prefab/脚本表达，高等级容易丢失低等级能力 | 已改为一个家族一个 Runtime Prefab/脚本，同实例原地升级 |
| 施工被建模为 LV0/施工 Prefab | 已改为 BuildingLifecycleStage.Construction |
| 普通升级替换对象，需要手写迁移人口、工人和进度 | 替换升级 API 已删除；状态保留在同一实例 |
| BuildingDefinition 与模块内联在等级 Prefab | 静态数据拆为 Family、ModuleSet、Presentation 独立资产 |
| 模块状态保存依赖硬编码类型或数组位置 | 有状态模块实现统一序列化接口，按稳定 ModuleId 保存 |
| 具体建筑与模块各有一套工人/生产/作物状态 | 模块成为唯一状态所有者；家族脚本只保留独有领域状态 |
| 家族、等级、施工和树木变体共用 BuildingId 语义 | 拆为 FamilyId、Stage、Level、StyleId、InstanceId |
| 等级数值与美术绑在 Prefab，缺图会阻塞玩法 | Runtime 与 View 分离；同样式低等级/统一占位回退 |
| Catalog 混合等级 Prefab 和建造条目 | Catalog 现在只登记 BuildingFamilyDefinition |
| 缺少结构验证，失效字段只能在运行时发现 | 新增 BuildingArchitectureValidator，校验家族、模块、等级、Prefab 和表现边界 |

## 仍然成立的研究结论

以下方向在终态中继续有效：

- 网格放置、占格事务、资源连接和 Overlay 应保持通用服务边界。
- BuildingBase 不应吸收居民房、市场、农田等家族特例。
- 跨家族复用能力应通过稳定模块和小接口组合。
- UI 只读取领域查询结果，不复制生产、升级和连接判定。
- 需要继续补 EditMode/PlayMode 自动化测试，重点覆盖施工原子扣费、原地升级状态保留、存档模块恢复、样式回退和批量放置回滚。

## 当前验证基线

2026-07-13 的 Unity 批处理结果：

- 8 个正式建筑家族；
- 8 个唯一 Runtime Prefab；
- 架构结构零错误；
- 23 条预期美术回填警告；
- 旧等级/施工 Prefab 目录和旧等级脚本已移除。

内容缺口（高等级数值、农田作物、捕鱼内容和美术资源）记录在《建筑说明》《建筑数值策划表》和《建筑Prefab与表现资源回填指南》中。

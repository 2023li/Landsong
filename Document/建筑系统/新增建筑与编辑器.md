# 新增建筑与编辑器

## 1. 先判断新增的是什么

| 需求 | 正确做法 |
| --- | --- |
| 同一玩法、不同外观 | 新增 Style，不新建家族 |
| 同一家族新增等级 | 正式 Excel 增加 Level 与现有 Module 配置 |
| 现有能力的新组合 | 新建 Family，复用 Module |
| 新的单一能力/状态 | 新增 Module；需要逐等级参数时再新增 LevelConfiguration |
| 完全不同占地、交互或身份 | 新建独立 Family 或独立系统 |

多数建筑不需要新脚本。仓库与粮仓复用 `workforce -> warehouse.operation`，树木与雕塑复用 `spatial_effect`，小土堆与小石堆复用 `harvestable.finite`，是标准案例。

## 2. 编辑器入口

打开：

```text
Landsong/Building/建筑编辑器
```

窗口一次创建并连接：

- `BuildingFamilyDefinition`
- `BuildingModuleSetDefinition`
- `BuildingPresentationDefinition`
- 统一结构的 Runtime Prefab
- `BuildingCatalog` 登记

窗口不会生成家族脚本，也不会把所有等级美术放入 Runtime Prefab。

## 3. 标准创建流程

1. 先在设计文档中明确 FamilyId、名称、分类、占地、蓝图、施工、等级和能力。
2. 在建筑编辑器选择“创建新家族”。
3. 填写稳定身份和固定占地。
4. 选择需要的模块模板并确认顺序。
5. 设置初始开放等级数量；它只创建 Level 结构，不代表最终数值已经完成。
6. 生成 Family、ModuleSet、Presentation 和 Runtime Prefab。
7. 在正式 Excel 补齐该 Family 的公共费用、等级和模块数值。
8. 使用建筑数值导表工具 Analyze 并导入。
9. 在编辑器的“编辑现有家族”检查结构、模块和表现映射。
10. 回填图标、放置预览、施工/运营 View。
11. 执行终态架构校验和 Play Mode 验证。

不要在窗口生成后直接手工修改 Excel 拥有的 SO 字段，否则下次导入会覆盖。

## 4. 关键字段

### 身份

- `FamilyId`：稳定英文 ID，例如 `building.granary`；存档和跨系统引用使用它。
- `DisplayName`：玩家可见名称，可本地化或修改。
- `Category`：建造菜单分类和部分专用交互判断。
- `IsDevelopmentCompleted`：未完成内容不能伪装成正式可玩建筑。

### 占地与地图

- Size 在家族所有等级固定，不允许升级改变占地。
- Required Terrain Keys：所有占地格必须具备的标签。
- Any Footprint Terrain Keys：完整占地中至少一格必须命中的标签。
- Movement Resistance / Action Power：道路和连接/范围相关规则，修改时同步阅读地图文档。

### 施工与等级

- Construction 是 Stage，不是 LV0。
- Initial Level Count 是创建多少个初始 `BuildingLevelDefinition` 槽位；例如 3 表示创建 LV1～LV3。
- `configured=false` 的等级不参与有效升级路径。
- 升级保留同一实例和全部 Module 状态。

### 表现

- Placement Preview 是独立字段。
- Construction View Mode 选择单一施工 View 或逐回合映射。
- Operational View 按 Level/Style 映射；缺高等级时可回退低等级。
- Style 只改变表现和建造栏图标，不复制玩法数据。

## 5. 模块配置在哪里

模块由两部分组成：

1. `BuildingModuleSetDefinition` 中的 Module 实例：声明这个家族拥有什么能力、默认值和执行顺序。
2. `BuildingFamilyDefinition.Levels[].Configurations` 中的 LevelConfiguration：声明某个等级如何重配目标 Module。

正式导入范围内的等级配置由 Excel 生成。窗口中的模块编辑器用于选择结构、查看模块名称和配置不由 Excel 管理的默认字段，不是第二套正式数值表。

配置与模块通过 C# 类型的 `Apply(BuildingBase)` 关联；稳定 `ModuleId` 用于唯一性、顺序和存档。不是让策划手工填写“某配置匹配某模块”的字符串。

## 6. 新增 Module

只有现有 Module 无法表达需求时才新增。

### 必要工作

1. 新建继承 `BuildingModuleBase` 的模块类。
2. 使用 `[BuildingModuleId("stable.id")]` 声明稳定 ID。
3. 选择所需生命周期接口，例如注册、施工、回合、点击、等级变化或拆除。
4. 对外提供能力接口，避免消费者判断模块具体类型。
5. 有运行时状态时实现 `IBuildingModuleStateSerializer`。
6. 若参数随等级变化，新建 `IBuildingLevelConfiguration`；若是全家族固定配置，不需要为形式统一额外建配置类。
7. 在编辑器的模块选择注册表、架构校验器配置映射中登记。
8. 需要策划导表时增加工作表 Row、读取、校验、差异分析和原子应用。
9. 更新本文、数值文档和相关能力文档。

模块和配置类不是永远一对一：只有“这个模块确实需要逐等级参数”时才需要 LevelConfiguration。

## 7. ItemDefinition 规则

模块与 LevelConfiguration 暴露给策划的物品字段统一使用 `ItemDefinition` 资产引用，不使用可编辑 ItemId 字符串。

ItemId 只用于：

- 库存服务 API；
- 运行时资源变化 DTO；
- 字典 Key；
- 存档。

正式 Excel 中仍可以使用稳定 ItemId，因为导表器会把它解析成 `ItemDefinition` 引用。解析失败必须阻止导入。

## 8. Runtime Prefab 固定结构

根节点必须有：

- 恰好一个 `BuildingBase`；
- `BuildingPresentationController`；
- 合法 Collider；
- 空 `ViewRoot`。

禁止包含：

- 施工/LV1～LVN 美术子节点；
- 家族派生建筑脚本；
- 玩法 Animator；
- 内联模块数组的第二份数据；
- 等级专属 Collider 或占地。

## 9. View Prefab

View Prefab 可以包含 SpriteRenderer、Animator、粒子、音频挂点和纯表现脚本，不得包含 `BuildingBase`、Collider、库存/岗位/生产脚本或任何玩法状态。

美术紧张时优先提供：图标、放置预览、一个施工 View、LV1 View。高等级可以先回退；以后替换只修改 Presentation。

## 10. 新增 Style

1. 在 Presentation 的 Styles 中填写稳定 StyleId、显示名和建造栏 Icon。
2. 为该 Style 配置放置预览和至少 LV1 View。
3. Family、ModuleSet、费用和等级不复制。
4. 放置请求与存档保存 StyleId。
5. StyleId 登记后保持稳定；只改显示名称不改 ID。

## 11. 编辑现有家族

“编辑现有家族”只应展示和修改当前选中 Family 的结构、ModuleSet 与 Presentation。若出现空白或巨大间距，先检查 Odin PropertyTree 是否在选择变化后正确重建、滚动容器是否嵌套错误，以及列表 Drawer 是否创建了不可见的高元素；不要在数据资产中添加占位元素来修 UI。

ModuleSet 列表必须显示模块中文名称和稳定 ModuleId；新增列表项后必须通过类型选择器创建有效 Module，不能留下不可选择的空托管引用。

## 12. 交付检查

- FamilyId 唯一且进入 Catalog。
- 分类、固定占地、地形、蓝图和菜单规则正确。
- ModuleSet 无空项、无重复 ModuleId，顺序满足依赖。
- 每个开放 Level 的 Configuration 能找到目标 Module。
- Runtime Prefab 结构唯一且 ViewRoot 为空。
- Presentation 的 Style/Level 矩阵与开放等级一致。
- 正式 Excel Analyze/Import 无错误、无待同步差异。
- `Validate Final Architecture` 结构错误为零。
- Play Mode 验证放置、施工、升级、回合、拆除和存读档。
- 更新 [建筑目录](建筑目录.md) 和必要的 [表现资源](表现资源.md) 清单。

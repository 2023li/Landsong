# 本地化重构设计

> 状态：已确认，阶段 A–E 已实施（所有决策采用推荐值）
> 调研日期：2026-07-18
> 范围：内置简体中文/英语、运行时切换、玩家自定义文本语言包、现有中文硬编码迁移

## 0. 实施结果

- 阶段 A：完成 Manager 拆分、稳定语言选项存档、`UI`/`Content`/`Gameplay` 表、任意 BCP 47 外部目录语言包与内置 fallback。
- 阶段 B：完成主菜单、设置、存读档、通用弹窗、游戏主面板和 501 个 TMP 固定文本绑定；语言切换由统一事件刷新动态 UI。
- 阶段 C：完成物品、物品组、建筑、地图、科技、任务、政策、远征、人才、继承等正式 Definition 的稳定 Key 接入。
- 阶段 D：完成游戏事件、建筑状态/详情、经济预测、科技、任务、远征、人才、继承等动态文本，以及随机任务的模板 Key + 稳定参数存档语义。
- 阶段 E：完成旧 `GameString` 移除、语言包模板导出、示例包/制作说明、运行时源码硬编码门禁、Prefab/Scene 门禁和自动化测试。
- 最终表项：`UI=595`、`Content=194`、`Gameplay=506`，内置 `zh-Hans`/`en` 均完整且 Smart String 参数集合一致。
- 自动化验证覆盖标准 CSV 多行/引号、重复 Key、非法 UTF-8、Unicode 稳定 Key；发布校验会阻止中英缺项、Smart 参数不一致和玩家文本硬编码回流。

## 1. 当前事实

### 1.1 已有基础

- 项目使用 Unity `6000.3.5f2`、Unity Localization `1.5.12` 和 Addressables `2.8.1`。
- 已创建 `zh-Hans`、`en` 两个 Locale，以及 `GameString` StringTable Collection。
- `GameLocalizationManager` 已经负责系统语言、用户选择、AppData 持久化、外部 CSV 扫描和运行时 StringTable 覆盖。
- 外部语言包目录由 `IOManager.ExternalLanguagePacksFolderPath` 提供，目前位于应用存档根目录下的 `ExternalLanguagePacks`。
- 文档已冻结“外部语言包只覆盖文本，不允许携带脚本或任意资源路径”的边界，本设计继续遵守该边界。

### 1.2 当前本地化实际尚未启用

- `GameString` 的 Shared Data 没有任何 Key，`zh-Hans`、`en` 两张表也没有任何条目。
- 自研代码中除 `GameLocalizationManager` 外，没有业务代码使用 `LocalizedString`、`GetLocalizedString` 或同类接口。
- Prefab/Scene/SO 中也没有发现实际的 StringTable Reference。
- `OnLanguageChanged` 当前没有任何订阅者，动态拼接文本不会因为切换语言而重新格式化。
- 现有 `ForceReloadAllLocalizeStringEvents` 会扫描全部对象并切换 `LocaleOverride`，但项目中目前没有真正接入表项的 `LocalizeStringEvent`，因此这段逻辑不能解决现有动态中文文本问题。

### 1.3 硬编码规模（启发式盘点，不是最终交付清单）

- C# 中约有 4,202 行包含中文字符串字面量；剔除 `Editor`、常见日志/异常、Inspector 特性后，仍约有 2,823 行可能属于运行时玩家可见文本。
- Prefab/Scene 中发现约 159 个包含中文的 TMP `m_text` 序列化值，分布在 38 个文件中；`Panel_Game.prefab` 一处就有 80 个。
- ScriptableObject 内容中，至少有 207 个 `displayName`、`description` 等显示字段，分布在 119 个文件中。
- 主要迁移热点是 UI、建筑状态与详情、游戏事件、任务、科技、库存物品、远征、人才、继承和经济预测。
- 上述数字包含部分编辑器提示、调试信息和非玩家文本；实施前需要由自动审计工具输出精确分类清单。

### 1.4 现有外部 CSV 方案的能力边界

当前外部 CSV 实际是“覆盖某个已存在 Unity Locale 的部分表项”，还不是完整的任意语言支持：

- `LanguageCode` 必须能在 Unity `AvailableLocales` 中找到；目前实际上只能可靠选择 `zh-Hans` 或 `en`。
- 没有独立的目标语言与回退语言概念，无法正确表示“日语文本，缺项时回退英语”。
- 没有格式版本、游戏 Key 集版本、作者、兼容性或校验结果。
- 重复 PackId、重复 Key、未知 Table/Key、缺失 Key 和 Smart String 参数不一致没有完整诊断。
- 自写 CSV 解析器不支持跨行字段，并会裁掉文本首尾空白。
- 语言包缺失时的恢复状态不完整；自定义语言码和内置来源可能形成不一致存档。
- 管理器直接替换全局 TableProvider、清空 TablePostprocessor，并依赖强制刷新技巧，责任过多且容易影响 Unity Localization 的其他扩展。

### 1.5 字体边界

当前主要 TMP 字体是单个动态字体资产，未配置 fallback font。内置中英文预计可覆盖，但“任意玩家语言”能否正常显示取决于该字体源文件是否包含对应字形。文本语言包不能自动解决阿拉伯文、泰文、日韩字符、复杂排版和字体授权问题。

## 2. 设计目标

1. 内置 `zh-Hans` 和 `en` 必须完整可用，并能在运行时可靠切换和持久化。
2. 逻辑 ID、存档和领域规则不依赖当前显示语言。
3. 玩家语言包只能提供文本；解析失败或内容缺失不能破坏内置语言。
4. 自定义语言可以声明目标 Locale 和一个内置回退 Locale，允许制作不完整语言包。
5. 静态 UI、动态格式文本、ScriptableObject 内容和历史/运行时消息都有明确的本地化方式。
6. 缺 Key、重复 Key、参数不匹配、硬编码回流和字形缺失可以被编辑器校验或自动测试发现。
7. 语言切换完成事件只在新语言资源真正可用后发出，所有动态 UI 能在同一刷新周期更新。

## 3. 推荐架构

### 3.1 责任拆分

保留现有全局 `GameLocalizationManager` 类型作为 AppBoot 和 UI 的序列化门面，不批量迁移这个已被 Unity 序列化引用的类型；新建的纯 C# 类型进入 `Landsong.Localization` 命名空间。

```text
AppBoot / 设置 UI
       |
       v
GameLocalizationManager（生命周期、选择、事件、公开查询）
       |
       +-- LanguagePackRepository（扫描和 PackId 去重）
       +-- LanguagePackReader（manifest / CSV 读取）
       +-- LanguagePackValidator（Schema、Key、Smart 参数、大小限制）
       +-- RuntimeStringTableProvider（构建并提供合并后的运行时表）
       +-- LocalizedTextResolver（动态文本与 Smart String 格式化）
       +-- LocalizationSelectionStore（与 AppData 的单一映射）
```

建议把当前文件中的 `LanguagePackInfo`、`ExternalLanguageTextPatch`、`RuntimeMergedStringTableProvider`、CSV 解析和存档映射拆到独立文件。Manager 自身只保留下列职责：

- 等待 Unity Localization 初始化；
- 枚举可用语言选项；
- 请求切换并返回成功/失败结果；
- 在成功后保存稳定选择并发布 `LocalizationChanged`；
- 对外提供当前 Locale、PackId 和诊断摘要。

不再让 UI 操作“当前预览下标”。UI 选择并提交稳定的 `LanguageOptionId`；下标只属于当前控件，不能进入 Manager 状态或存档。

### 3.2 语言选择存档

建议把存档语义收敛为：

```text
Mode: System | Explicit
SelectedLanguageOptionId: builtin.zh-Hans | builtin.en | <external PackId>
```

- 外部语言包以 `PackId` 为稳定身份，不以文件名为身份。
- LocaleCode、DisplayName、FallbackLocaleCode 都从当前安装语言包读取，不重复写入 AppData。
- 外部包被移除或失效时，明确回退到项目默认内置语言并保存该结果。
- 项目仍处于公开版本前，可直接提升 AppData/GameData 版本，不增加旧结构兼容链。

### 3.3 StringTable 划分

建议由一个空的 `GameString` 逐步改为三个职责稳定的 Collection：

| Table | 内容 |
| --- | --- |
| `UI` | 菜单、按钮、面板标题、固定提示和设置项 |
| `Content` | 建筑、物品、地图、科技、任务、政策、远征、人才等定义的名称与描述 |
| `Gameplay` | 状态、事件、错误原因、动态格式、回合/数量/概率等 Smart String |

不建议按每个面板创建一张表；表过多会增加外部语言包维护和运行时装载成本。Key 使用小写英文分层语义：

```text
ui.main_menu.new_game
ui.settings.language
content.item.wheat.name
content.item.wheat.description
content.technology.scientific_theory.name
gameplay.expedition.success_chance
gameplay.quest.category.mainline
gameplay.building.status.road_unreachable
```

规则：

- 显示出来的中文、Unity 对象名、文件名和枚举成员名都不能作为稳定 Key。
- Content Key 优先从正式稳定 ID 推导，例如 `ItemId`、`FamilyId`、`TechnologyId`、`QuestId`。
- 删除/重命名 Key 必须经过审计；语言包以 Key 为契约。
- `zh-Hans` 作为初始内容来源，`en` 是必须完整的内置翻译；两者缺项都应阻止发布构建。

### 3.4 不同文本类型的接入方式

#### 静态 Prefab 文本

- 使用 `LocalizeStringEvent` 或项目封装的轻量绑定组件。
- Prefab 只保存 Table/Key，不再把中文作为运行时默认值。
- 空文本、数字占位和仅编辑器标识不强制本地化。

#### 动态 UI 文本

- 通过 `LocalizedTextResolver` 使用 Key + 命名参数格式化。
- 使用完整 Smart String，不拼接多个可翻译片段。例如把“预计成功率”“额外收益”和百分号作为一条可重排文本。
- 动态面板订阅统一的 `LocalizationChanged`，重新从领域状态渲染；不缓存最终显示字符串。
- 数字、日期、百分比按目标 Locale 的 CultureInfo 格式化。

#### ScriptableObject / 正式内容

- 领域 Definition 保存稳定本地化引用或可推导 Key，不把当前语言字符串当作领域数据。
- UI 层解析名称和描述；领域服务的排序、相等判断和存档只使用稳定 ID。
- `DisplayName` 这类当前返回中文字符串的 API 应逐步改成 `DisplayNameKey`/`NameText` 等明确语义，避免领域层静默读取当前语言。
- 建筑/库存 Excel 导入器只维护稳定 ID、数值和本地化 Key；不再把中文显示名反向写成唯一运行时文本。
- 内置翻译的唯一权威源建议继续使用 Unity StringTable 资产；为翻译人员提供受控 CSV 导入/导出工具，不再维护第二套正式翻译源。

#### 运行时消息、状态与随机内容

- `GameEventMessage`、建筑状态、失败原因、预测说明等改为 `TextKey + serializable arguments`，渲染时再格式化。
- 参数尽量保存稳定 ID 或数值；物品、科技、建筑名称在渲染时按当前语言解析。
- 随机任务不能把已格式化中文名称/描述写进存档。存档应保存模板 Key 与生成参数，加载或切换语言后重新生成显示文本。
- 玩家输入的存档名、角色名等用户内容保持原文，不进入翻译表。

#### 编辑器与日志

- Inspector Label、开发者日志、资产名和编辑器校验信息不属于首轮玩家本地化范围。
- 会直接出现在正式玩家 UI 的错误信息必须本地化；仅写入 Console 的诊断可以继续使用中文或英文。

### 3.5 切换语言的数据流

```text
用户选择 LanguageOptionId
  -> 解析内置 Locale 或外部 manifest
  -> 校验语言包并准备运行时表
  -> 外部新语言：创建 Runtime Locale，设置内置 fallback
  -> 安装 RuntimeStringTableProvider
  -> 释放相关旧表缓存
  -> 设置 LocalizationSettings.SelectedLocale
  -> 等待必要表可用
  -> 保存选择
  -> 发布 LocalizationChanged
  -> 动态 UI 按当前领域状态重绘
```

静态 `LocalizeStringEvent` 应由 Unity 的 Locale 变化自动刷新。现有扫描全场景并临时切换 `LocaleOverride` 的强制刷新方案应在正确接入后删除；不能把它作为动态 UI 的通用刷新机制。

## 4. 玩家语言包 V1 规范

### 4.1 推荐载体

首版建议使用“一包一目录”，比当前顶层单 CSV 多一个 manifest，但无需解压、没有 Zip Slip 风险，也便于玩家修改和排错：

```text
ExternalLanguagePacks/
  community.ja/
    manifest.json
    strings.csv
```

后续如果需要发布到社区平台，可以在不改变包内结构的前提下增加只读 `.landlang` ZIP；不建议首版同时维护两种读取路径。

### 4.2 manifest 示例

```json
{
  "schemaVersion": 1,
  "packId": "community.ja",
  "displayName": "日本語",
  "localeCode": "ja-JP",
  "fallbackLocaleCode": "en",
  "author": "Example Author",
  "description": "Japanese community translation",
  "targetKeysetVersion": 1
}
```

字段语义：

- `schemaVersion`：文件格式版本；不支持时拒绝载入。
- `packId`：稳定且全局唯一，只允许安全的 ASCII 标识字符。
- `displayName`：语言列表中显示的语言自称。
- `localeCode`：用于 CultureInfo、复数和排版规则的 BCP 47 语言标签。
- `fallbackLocaleCode`：只允许指向项目声明支持的内置 Locale，首版为 `en` 或 `zh-Hans`。
- `targetKeysetVersion`：用于提示语言包是否落后；默认只警告，不因为少量缺 Key 阻止加载。

### 4.3 strings.csv

```csv
Table,Key,Text
UI,ui.main_menu.new_game,ニューゲーム
Content,content.item.wheat.name,小麦
Gameplay,gameplay.expedition.success_chance,成功率は {chance:0.#}% の見込み
```

- UTF-8，支持标准 CSV 引号、逗号、换行和双引号转义。
- 外部包可以是部分翻译；缺项使用 manifest 指定的内置语言回退。
- 重复 `Table + Key` 是错误；未知 Table/Key、缺 Key 和 Keyset 版本不一致是可见警告。
- Smart String 的参数名、参数类型约束和大括号结构必须与基础条目兼容，否则该条目不生效并回退。
- 空字符串与缺失条目语义不同：空字符串表示译者明确希望显示为空；缺失才触发回退。

### 4.4 安全与资源限制

- 只读取固定名称的 manifest 和 CSV，不接受脚本、DLL、Addressables、任意资源路径或远程 URL。
- 校验单包大小、最大条目数、单条文本长度、PackId/LocaleCode、重复键和非法路径。
- 读取失败只使该包进入 Invalid 状态，不能修改内置表或当前成功应用的语言。
- 运行时表在内存中创建并标记 `DontSave`，切换时统一释放。
- 第一版不允许语言包携带字体。可选语言受游戏内置字体和 TMP fallback 配置支持范围限制。

## 5. 编辑器工具与质量门禁

建议增加一个本地化审计/迁移窗口，而不是纯手工替换：

1. 扫描 Prefab/Scene 中玩家可见 TMP 硬编码文本。
2. 扫描运行时代码中的疑似玩家可见字符串，并允许显式标记日志/编辑器例外。
3. 扫描 Definition 的名称/描述并按稳定 ID 建议 Key。
4. 检查所有 Table 的 `zh-Hans`/`en` Key 对齐、空翻译和重复语义 Key。
5. 导出玩家语言包模板及 Key 说明。
6. 校验 Smart String 参数集合和 TMP 富文本标签。
7. 可选加入 Pseudo Locale，用于发现文本截断、布局溢出和未接入本地化的 UI。

发布前最低门禁：

- 内置 `zh-Hans`、`en` 无缺 Key；
- 所有 Smart String 可解析且参数集合一致；
- 正式 Prefab/Scene 不存在未登记的玩家可见固定文本；
- 外部包异常不会中断启动或切换；
- 连续切换 `zh-Hans -> en -> 外部包 -> zh-Hans` 后无旧语言残留；
- 重启后语言选择一致，删除当前外部包后执行确定性回退；
- 核心 UI 在英文和伪本地化下无严重截断；
- 字体审计能报告目标语言缺失字形。

## 6. 推荐实施顺序

### 阶段 A：基础设施与一个纵向切片

- 拆分 Manager 责任并定义新存档选择模型。
- 建立 `UI`、`Content`、`Gameplay` 表和 Key 规范。
- 实现外部目录语言包 V1、校验结果和运行时 Locale/fallback。
- 完成“主菜单 + 设置语言项 + 一个游戏内面板 + 一个 Definition + 一条动态 Smart String”的完整中英/外部包切片。
- 验证多次切换、重启、包删除和缺项回退后，再批量迁移。

### 阶段 B：应用壳层与高频 UI

- 主菜单、设置、存读档、通用按钮、弹窗和 `Panel_Game` 固定文本。
- 建立动态 UI 的统一语言变化刷新方式。

### 阶段 C：正式内容 Definition

- 物品/物品组、建筑、地图、科技、任务、政策、远征、人才、继承。
- 同步调整建筑和库存导表器，保证本地化 Key 与正式稳定 ID 一致。

### 阶段 D：动态消息与存档语义

- 游戏事件、建筑状态、失败原因、经济预测、任务进度、随机任务和所有动态拼接文本。
- 把存档中的最终显示字符串改为模板 Key + 参数或稳定 ID。

### 阶段 E：硬编码清零与发布门禁

- 完成审计白名单，修复遗漏，加入 EditMode/PlayMode 自动测试和语言包制作说明。
- 更新 `Document/音频与本地化/README.md` 并把确认后的结论纳入正式架构文档。

## 7. 不建议的方案

- 不建议继续把所有功能堆进当前 `GameLocalizationManager`。
- 不建议只替换 Prefab 固定文字而保留代码动态拼接；切换语言后仍会大量残留中文。
- 不建议把中文原文直接当 StringTable Key；翻译修改会破坏引用和玩家语言包。
- 不建议让 Definition/Service 到处同步调用 `GetLocalizedString()`；本地化应在展示边界解析，领域逻辑只持有稳定 ID/Key。
- 不建议把已格式化的当前语言文本写入长期存档。
- 不建议首版允许玩家包携带字体、Prefab、图片或脚本。
- 不建议为了保留当前单 CSV 结构增加兼容分支；项目仍处于开发阶段，可以直接定义正确的 V1 包规范。

## 8. 确认结论

以下决策均已确认并据此完成实施：

1. **自定义语言范围**：推荐支持任意合法 BCP 47 Locale，而不只是中英文本变体；每个外部包必须选择 `en` 或 `zh-Hans` 作为缺项回退。
2. **语言包载体**：推荐首版使用“一包一目录 + `manifest.json` + `strings.csv`”，暂不支持旧顶层单 CSV 和 ZIP。
3. **项目默认回退语言**：推荐正式版本使用 `en`；开发期仍要求 `zh-Hans` 和 `en` 都完整，不依赖回退掩盖漏翻。
4. **不完整玩家翻译**：推荐允许加载，语言列表显示缺失/未知 Key 数量；只拒绝结构错误、重复 Key、非法 Smart String 和越界内容。
5. **字体与资源**：推荐首版语言包严格只含文本。需要支持哪些中英以外的目标文字系统，应先据此补充游戏内置 TMP fallback 字体。
6. **内置翻译权威源**：推荐 Unity StringTable 资产为唯一正式翻译源，通过编辑器工具导入/导出 CSV；建筑/库存 Excel 只保存稳定 ID、数值和本地化 Key。

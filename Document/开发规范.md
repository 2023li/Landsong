# Landsong 开发规范

> 本文是项目级约束。系统细节以对应目录的 `README.md` 为准。

## 1. 事实与修改顺序

修改前按以下顺序确认事实：

1. 当前 C# 领域契约与真实调用链。
2. 正式配置源或 Unity 资产中的当前数据。
3. Prefab/Scene 的序列化绑定。
4. 对应校验器、导表器和运行时行为。
5. 开发阶段不用兼容任何旧接口和存档
6. 文档。

文档与实现冲突时先以实现为准，并在同一任务修正文档。不要凭旧类名、历史 README 或截图补一条新的兼容路径。

### 1.1 开发阶段禁止临时修补

项目处于开发阶段，不允许用临时修补代替正确设计。问题暴露出责任边界、数据模型、事件来源或生命周期错误时，必须修正正式入口和完整调用链，不得只让当前案例“看起来能运行”。

禁止：

- 为“最后一次”“第三次”或某个特定家族增加特殊分支，掩盖通用模型仍然依赖已销毁对象的问题；
- 为保留错误依赖新增兼容开关、回退路径、延迟一帧、重复状态或同义 API；
- 通过吞异常、运行时补组件、按名称搜索或静默降级掩盖 Prefab、资产或服务配置错误；
- 只修改调用链末端的 UI/表现，而不修正实际缺失的领域事件、事务结果或状态所有权；
- 在没有用户明确批准的情况下引入计划以后删除的临时代码。

如果正确修复需要扩大范围，应先说明错误边界和必要范围；无法在当前任务安全完成时停止并请求决策，不写入临时方案。美术资源尚未交付时，经过明确约定保留的空 View 节点或占位资产属于内容缺口标记，不得改变玩法规则，也不构成运行时代码修补。

## 2. 领域入口

运行时服务统一通过：

```csharp
GameSystem.Instance.Services
```

可访问的主要服务包括：

- `Inventory`
- `EconomyForecast`
- `Turn`
- `Dynasty`
- `Buildings`
- `Events`
- `Technology`
- `Policies`
- `Quest`
- `Expeditions`
- `Talents`
- `Inheritance`
- `BuildingBlueprints`
- `BuildingSelection`
- `BuildingCatalog`
- `TechnologyUnlockContents`
- `GlobalBuffs`

例如：

```csharp
GameSystem.Instance.Services.Quest.TrySubmitResources(questId);
GameSystem.Instance.Services.Technology.TryStartResearch(technology);
GameSystem.Instance.Services.Talents.TryRefreshOffers(out var result);
```

`GameSystem` 只新增真正的游戏级编排：回合顺序、游戏结束、跨服务协调和服务装配。不要重新增加任务、科技、远征、人才、继承或建筑蓝图的同义公开代理。

## 3. 状态所有权

- 一个状态只能有一个领域所有者。UI、预测器和调试工具读取状态，不持有第二份真相。
- Service 持有领域集合与事务；建筑 Module 持有单个建筑实例能力状态；ScriptableObject 持有静态定义。
- 跨系统查询优先使用公开服务或能力接口，不按具体 `MonoBehaviour` 类型分支。
- 回合内资源归因、统一结算和跨建筑聚合进入 `TurnService` 或更高层编排，不放到某个建筑的事后补丁中。
- 范围效果、库存槽位、资源提供点等建筑能力通过接口/Provision 暴露，消费者不识别家族名。

## 4. UI 与输入

- 面板只负责订阅、格式化、实例化预制体和转发用户操作。
- 运行时 UI 元素应由 Prefab 预先创建并通过 `[SerializeField]` 绑定；不要按层级名称搜索，也不要临时创建 TMP/Image 来掩盖缺失引用。
- 所有库存格统一使用正式槽位 Prefab；不同类型的 Root 映射属于 UI，不进入数值表。
- 指针、触摸、UI 命中和相机输入阻断统一经过 `InputController`。
- 双击、长按等全局手势时间使用 `Landsong.InputSystem.InteractionConstants`，不得在业务组件中再次写 `0.3f`、`0.45f` 等同义阈值。
- 与具体组件视觉相关且需要美术调整的持续时间可以保留为序列化字段，例如淡入淡出、浮字间隔；它们不属于全局手势判定。
- 飘字、伤害数字等一次性反馈在接收领域事件时立即解析为位置、图标、文本和来源 ID 等纯显示快照；队列与动画实例不得持续持有或查询 `BuildingBase` 等领域对象。领域对象随后移动、禁用或销毁，不影响已经发生的反馈。

## 5. 数据与 ID

- 显示名称可以修改；存档、字典、跨资产引用使用稳定 ID。
- 策划配置中的物品字段使用 `ItemDefinition` 资产引用；仅库存 API、DTO、字典键和存档使用派生 `ItemId`。
- 建筑使用稳定 `FamilyId`、`ModuleId`、`EffectId`；科技使用 `TechnologyId`；地图使用 `MapId`。
- 不用文件名、Hierarchy 名称或数组下标作为运行时身份。
- 正式 Excel 导入字段只在 Excel 维护；导入后的 SO 是生成结果。Unity 拥有的结构和表现字段不得写回 Excel。

## 6. 存档

统一保存入口：

```csharp
DataManager.Instance.SaveCurrentGame();
DataManager.Instance.SaveCurrentGame(GameDataSaveMode.NewSave);
```

统一存档列表：

```csharp
DataManager.Instance.GameDataMetaList
```

各服务实现自身 `CaptureSaveData` / `RestoreSaveData`，由 `GameRuntimeSnapshotService` 编排。运行时对象引用不直接序列化，使用稳定 ID 和轻量引用结构恢复。

首个公开版本前，`GameSaveRepository` 只接受 `GameData.CurrentDataVersion` 完全一致的存档。结构变化提升项目级版本，子系统不得增加半兼容迁移；正式规则见 [架构决策](架构决策.md)。

## 7. 事件

- 领域状态变化使用强类型 C# 事件，例如 `InventoryChanged`、`TechnologyUnlocked`、`BuildingsChanged`。
- `GameEventService` 保存的是给玩家显示的游戏事件消息，不是替代领域事件的万能 EventBus。
- 订阅者必须在禁用/销毁时解除订阅；恢复流程应避免在半恢复状态发送完整刷新事件。
- 跨系统协调由 `GameSystem` 订阅服务事件，不让两个领域 Service 互相直接持有具体实现。

## 8. 编辑器与正式配置

- 建筑：使用 `Landsong/Building/建筑编辑器`、建筑数值导表工具和终态架构校验器。
- 库存：使用 `Landsong/Inventory/库存数值导表工具`。
- 地图：使用 `Landsong/地图/地图目录与校验`。
- 科技：使用 `TechnologyEditorWindow`；改 ID、依赖或模板后重新生成科技树 UI。
- 导入必须先 Analyze，再原子应用；有错误时不允许部分写入。
- 编辑器代码放在 `Assets/Landsong/Scripts/Editor`，运行时程序集不得引用 `UnityEditor`。

## 9. 命名空间与序列化

新建的 C# 类型必须进入对应 `Landsong.*` 命名空间。现有全局命名空间的 `MonoBehaviour`/`ScriptableObject` 不批量迁移，因为类全名变化会影响 Unity 序列化引用；该边界已经在 [架构决策](架构决策.md) 中冻结。

不要为了目录整齐直接改已序列化组件的命名空间。若业务必须迁移，必须建立专门任务，使用 Unity 支持的类型迁移策略并验证全部 Scene、Prefab 和资产。

## 10. 验证与交付

1. 先做与风险相称的源码编译。
2. 检查 Prefab/Scene/SO 的真实序列化引用。
3. 执行系统专用导表器和校验器。
4. 涉及生命周期或存档时做 Play Mode 新建、保存、退出、加载。
5. 更新对应系统 README 和必要子文档。
6. 检查 Markdown 相对链接和旧路径引用。
7. 明确说明哪些已由编译/校验器证明，哪些仍需要 Unity 运行时验收。

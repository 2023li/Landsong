# Landsong AI 开发原则

这是项目级的 AI 协作入口，不是某个系统的实现手册。先在这里确认通用原则与负责模块，再进入该模块的详细规则文档。

当文档、历史讨论与仓库当前代码或 Prefab 冲突时，以当前实现为准，并在任务完成时修正文档。

## 使用顺序

1. 阅读本文件，确认通用原则与任务所属模块。
2. 阅读对应模块的入口规则。
3. 按模块规则读取必要的代码、Prefab 和数据资产。
4. 完成代码、资产与文档更新，并以适当方式验证。

不要跳过第 2 步，直接凭类名或旧讨论修改代码。

## 通用原则

- 不需要兼容任何存档数据、过时API
- 以当前仓库代码、Prefab、Scene 和 ScriptableObject 为事实来源。
- 发现文档过期或与实现不符时，修正文档；不要在新代码中延续错误描述。
- 先沿真实调用链定位责任，再决定修改点；不要用 UI 或临时补丁掩盖领域逻辑问题。


## 建筑系统开发

负责建筑的 AI / 开发者必须按以下顺序阅读：

1. [建筑系统交接总览](建筑系统/README.md)：阅读顺序、交接状态和权威边界。
2. [AI_添加建筑规则.md](建筑系统/AI_添加建筑规则.md)：入口文件、职责边界、修改约束、Prefab/存档交付清单。
3. [建筑扩展规则README.md](建筑系统/建筑扩展规则README.md)：终态领域模型、状态所有权和扩展边界。
4. [建筑说明.md](建筑系统/建筑说明.md) 与 [建筑数值策划表.md](建筑系统/建筑数值策划表.md)：当前家族、数值、模块和内容缺口。
5. 涉及美术、动画或缺资源回退时阅读 [建筑Prefab与表现资源回填指南.md](建筑系统/建筑Prefab与表现资源回填指南.md)。
5. 涉及占地、地形、资源连接、放置预览、初始建筑或空间 Buff 时阅读 [AI_地图系统.md](AI_地图系统.md)。
6. 与任务直接相关的 Family、ModuleSet、Presentation、Runtime/View Prefab 和脚本。

新增或维护建筑优先使用 Unity 菜单 `Landsong/Building/建筑编辑器`，操作规范见 [建筑编辑器窗口规划与使用.md](建筑系统/建筑编辑器窗口规划与使用.md)。建筑修改完成后必须执行 `Landsong/Building/Validate Final Architecture`。每级 Runtime Prefab、等级脚本、Prefab 替换升级和巨型内嵌美术 Prefab 均为禁止形态。

## 地图系统开发

负责地图、格子规则、地图加载、初始建筑或 Overlay 的 AI / 开发者必须按以下顺序阅读：

1. [AI_地图系统.md](AI_地图系统.md)：当前架构、事实来源、运行链路、扩展边界、并行维护约束和排错入口。
2. [开发_创建地图.md](../开发_创建地图.md)：策划在 Unity Editor 中创建和验收地图的操作步骤。
3. [README_地图系统与建筑放置重构.md](README_地图系统与建筑放置重构.md)：本轮重构的完整决策记录和历史实施规格。
4. 与任务直接相关的 Scene、ScriptableObject、Prefab 和脚本。
